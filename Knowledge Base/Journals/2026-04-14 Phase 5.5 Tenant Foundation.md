# 2026-04-14 Phase 5.5 — Tenant Foundation

**Confidence Level:** High

---

## Roadmap Jump Declaration

This work is a **deliberate, conscious Phase 5.5 extension** — not accidental scope drift.

The Phase 4 roadmap states: *"No self-service tenant onboarding at MVP — platform admin onboards tenants manually."*

Phase 7 lists: *"Multi-council onboarding self-service tooling."*

**Decision:** Pull forward the tenant foundation (not self-service onboarding) now, because:
1. The subdomain-per-tenant model was chosen and requires schema + resolver work before any other tenant can be added
2. The current `X-Tenant-Slug` header approach is a temporary bodge that would have to be removed anyway
3. Building this now costs less than retrofitting it later against a larger codebase

**What was NOT built:** Self-service onboarding, SSO, billing, deletion pipelines. Those remain Phase 7.

---

## Decisions Made

### 1. Tenant resolution — dual-mode

**Decision:** Centralise all tenant resolution in `src/lib/tenant-resolver.js`. Two strategies, tried in order:

1. **Subdomain from Host header** — production model. `riverside.zanflow.co.uk` → extract `riverside` → lookup by `tenants.subdomain`
2. **X-Tenant-Slug header fallback** — workers.dev / local dev only. Required because `workers.dev` does not support wildcard subdomains.

The fallback is isolated to one file and is clearly marked for removal. When the platform moves to a real domain, delete the fallback block in `tenant-resolver.js` only. No other file changes.

**Rationale:** Do not duplicate tenant resolution logic across routes. One module, one place to change.

### 2. Tenant status model

**Decision:** Seven states added to `tenants.status`:
```
pending_verification → trial → active → suspended → expired → scheduled_deletion → deleted
```

Status is enforced at the resolver level — `suspended`, `expired`, `scheduled_deletion`, and `deleted` tenants are blocked from all API traffic. No tenant-scoped handler needs to re-check status.

**Existing Riverside tenant** seeded as `active`.

### 3. Tenant limits

**Decision:** `tenant_limits` table created — one row per tenant. Defaults: `max_staff_users = 3`, `max_applications = 50`. These are trial defaults.

Limits enforced server-side at:
- `POST /api/applications` — checks `max_applications` before insert
- `POST /api/platform/tenants/:id/admin` — checks `max_staff_users` before user creation

Riverside Council seeded with generous limits (100 staff, 10000 applications) as it is an active internal tenant.

### 4. Platform admin tenant management routes

**Decision:** Internal-only endpoints added at `/api/platform/tenants/*`. Require `is_platform_admin = true` on staff session. No frontend built yet — these are API-only for now.

Endpoints:
- `GET  /api/platform/tenants` — list all tenants with counts
- `POST /api/platform/tenants` — create new tenant with subdomain, status, limits
- `GET  /api/platform/tenants/:id` — get one tenant
- `PUT  /api/platform/tenants/:id/status` — lifecycle state transitions
- `POST /api/platform/tenants/:id/admin` — create first tenant admin user

All mutations audited via `writeAuditLog`.

### 5. Subdomain reservation

**Decision:** A `RESERVED_SUBDOMAINS` set is maintained in `tenant-resolver.js` (for resolution-time blocking) and mirrored in `platform-admin.js` (for creation-time validation). Reserved words include: `www`, `api`, `admin`, `platform`, `app`, `mail`, `billing`, `staging`, `dev`, `test`, etc.

Format constraint: `^[a-z0-9][a-z0-9\-]{0,61}[a-z0-9]$` — enforced both in DB (CHECK constraint) and application layer.

## Schema Changes (Migration 0009)

- `tenants.subdomain` — TEXT UNIQUE, format-validated
- `tenants.status` — lifecycle state machine, CHECK constraint
- `tenants.contact_name`, `contact_email` — onboarding contact
- `tenants.trial_started_at`, `trial_ends_at`, `activated_at`, `suspended_at`, `expired_at`, `scheduled_deletion_at`, `deleted_at`, `deletion_reason` — lifecycle timestamps
- New table: `tenant_limits` (tenant_id PK FK, max_staff_users, max_applications)
- Index: `idx_tenants_subdomain` (unique partial), `idx_tenants_status`
